# 调试推理规则

本模块定义调试/修复/回归排障任务的通用问题定位方法，目标不是记忆单一案例，而是帮助 agent 更快定位“第一处分叉”和真正拥有问题的数据/产物层。

---

## 适用场景

```yaml
适用:
  - bugfix / debug / 排障 / 回归问题
  - 运行时正常但历史/预览/回放异常
  - 原始数据正确但缓存/快照/渲染/派生产物错误
  - 接口 200 但语义错误、展示错误、分页错误、字段错误
  - 重新执行/重新讲/重试后才触发的问题

不适用:
  - 纯新增功能且无现象性缺陷
  - 纯文案修改
```

---

## 核心原则

### 1. 找第一处分叉，不找最后一个报错点

```yaml
目标: 定位“从哪里开始和正常路径不同”，而不是停留在最终报错页面、最终坏图、最终空数据
做法:
  - 同时画出 正常链路 与 异常链路
  - 标出两条链路共享的最后一个检查点
  - 标出两条链路首次不一致的步骤、字段、产物或缓存键
输出:
  - Last Known Good（最后一个确定正常的点）
  - First Bad Divergence（第一处明确异常的点）
```

### 2. 区分表示层，不把“字段存在”当成“语义正确”

```yaml
对每个问题都先识别涉及的表示层:
  - Source: 原始输入/上游稳定数据
  - Derived: 解析、拼接、渲染、裁剪、聚合、补全后的派生产物
  - Persisted: 数据库存档、缓存、快照、历史记录
  - Transport: 接口响应、消息体、事件流
  - Presentation: 客户端页面、预览图、历史卡片、播放页

规则:
  - 传输成功(HTTP 200 / code=00000) ≠ 语义正确
  - 字段非空 ≠ 字段内容正确
  - 页面异常 ≠ 页面就是根因层
  - 持久化记录异常时，优先检查写入时刻，不要只盯读取接口
```

### 3. 用不对称现象定位责任层

```yaml
当输出“部分正常、部分异常”时，优先利用不对称信息定位：
  - 上半坏下半好 → 检查组合顺序、局部素材源、局部渲染路径
  - 第一页坏后续页好 → 检查分页边界、首屏特殊逻辑、默认值
  - 仅重新执行/重新讲后坏 → 检查 null→非null、首次写入→复用旧记录、重试分支
  - 实时页面正常/历史页面异常 → 检查实时链 vs 写库/回放链
  - 文本正常/图片坏 → 检查图源选择、图片代理、渲染、缓存失效
```

### 4. 优先复用稳定产物，不优先重算

```yaml
若系统同时存在:
  - 上游已提供的稳定产物（如已生成图片、稳定快照）
  - 本地即时重算/重渲染/重拼接能力

优先级:
  先确认稳定产物是否可直接复用
  仅在稳定产物缺失/不可用时再重算

原因:
  本地重算路径通常更脆弱，依赖更多隐式格式、路径拼装、模板或缓存前提
```

### 5. 修复应落在分叉点拥有者，而不是症状层

```yaml
优先级:
  1. 修“第一处分叉”的拥有者
  2. 修错误的优先级选择或错误的 fallback
  3. 修错误的格式归一化/路径拼装/缓存键
  4. 最后才考虑在展示层打补丁

禁止:
  - 因页面报错就只改页面
  - 因接口字段异常就只改返回 DTO，而不追写入来源
  - 用额外兼容兜底掩盖更早的错误决策
```

## 执行步骤

### 步骤1: 定义症状与边界

```yaml
必须明确:
  - 用户可见症状
  - 触发条件
  - 不触发时的对照条件
  - 最近变化（重新讲、重试、切 tab、翻页、缓存命中、字段从 null 变非null）
```

### 步骤2: 绘制表示层地图

```yaml
对当前问题画出最小数据流:
  输入/上游 → 清洗/归一化 → 派生/渲染 → 持久化/缓存 → 接口返回 → 客户端展示

每层回答 4 个问题:
  - 谁生成
  - 谁拥有
  - 谁消费
  - 哪一层已经被证实正常/异常
```

### 步骤3: 对比正常链路与异常链路

```yaml
优先找一个“正常样本”与“异常样本”做并排对比
对比对象优先级:
  - 运行时实时链路 vs 历史回放链路
  - 初次执行 vs 重新执行
  - 旧实现 vs 新实现
  - 稳定上游字段 vs 本地派生字段
```

### 步骤4: 定位第一处分叉

```yaml
常见分叉类型:
  - 字段优先级变更
  - fallback 覆盖稳定值
  - 路径归一化规则变弱
  - 裸数据与代理路径的转换丢失
  - 写库字段与展示字段不一致
  - null→非null 后触发类型溢出/格式错误
  - runtime 使用文本，history 使用预渲染图
```

### 步骤5: 设计修复

```yaml
修复优先级:
  - 恢复稳定值优先级
  - 修正错误的派生逻辑
  - 修正错误的归一化/格式转换
  - 仅在必要时增加展示层容错

修复说明必须回答:
  - 为什么这个点是第一处分叉
  - 为什么改这里比改症状层更稳
  - 为什么不会破坏正常链路
```

### 步骤6: 设计回归验证

```yaml
回归至少覆盖:
  - 正常样本仍正常
  - 异常触发条件被命中时不再复现
  - 第一处分叉相关的边界条件（如 null→非null、重新讲、缓存命中、重试）
  - working path 与 failing path 的对照断言
```

---

## 高价值检查清单

```yaml
做 bugfix/debug 时，优先检查:
  - 是否存在“一个稳定源 + 一个本地派生源”并被错误选择
  - 是否存在“写入链”和“读取链”使用不同字段
  - 是否存在“实时显示”和“历史回放”使用不同表示层
  - 是否存在“部分正常/部分异常”的不对称现象可用于定位
  - 是否存在“接口成功但业务语义错误”的伪健康状态
  - 是否存在“旧实现只在缺失时重算，新实现无条件重算”的回归
  - 是否存在“路径/尺寸/代理前缀/格式元数据”在新实现里被弱化
```

---

## 输出要求

```yaml
调试/修复任务的分析摘要至少包含:
  - 症状
  - 正常链路
  - 异常链路
  - 第一处分叉
  - 分叉拥有者
  - 修复策略
  - 回归验证点
```
